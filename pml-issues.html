P0 · DATA LOSS

SQS Messages Permanently Lost

17 SQS consumers delete messages BEFORE processing. Any failure = money moves but records don't update.

// eq-fms-consumers — EVERY SQS consumer follows this pattern: Session session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE); // Message deleted on receive! @Override public void onMessage(Message message) { // Message is ALREADY DELETED from SQS at this point try { processPayment(message); // If this throws... } catch (Exception e) { log.error("[eventMessage] [Exception]", e); // ...message is gone forever } }

Settlement Comms

Settlement notifications vanish

Block Transactions

Fund block status disappears

Pay-in Clearance

Payment clearance not recorded

TechExcel Sync

BO transaction sync lost

Fix: Change to Session.CLIENT_ACKNOWLEDGE and call message.acknowledge() only AFTER successful processing. Add DLQs for all queues.

RELIABILITY

Silent Failures & Missing Safeguards

Bugs that fail without anyone knowing. No circuit breakers. No DLQs. No monitoring on payment consumers.

Zero Monitoring on Financial SQS

None of the 7 payment SQS consumers have:

  • × Micrometer counters/timers
  • × Messages received/processed/failed metrics
  • × Processing latency tracking
  • × Dead letter queues
  • × Alerting on failures

Kafka: Auto-Commit + Swallow

eq-fms-consumers Kafka factory defaults auto.commit=true. Offset committed before processing. Exception caught and swallowed. Messages permanently lost.

Zero Circuit Breakers

Only payments (deprecated Hystrix) and bo-common (outdated Resilience4j 1.3.1). All other services have none.

More Silent Failures

  • Retry exhaustion drops messages in notification service (no DLQ)
  • Cron job error kills scheduler thread (dnd-aggregator, cache reload)
  • Notification service: 3 threads for 20+ Kafka listeners + cron jobs
ARCHITECTURE

Structural Design Flaws

Single points of failure, vendor lock-in, unbounded queries, timezone bugs

RupeeSeed Lock-in

22 files depend on RupeeSeed. No abstraction, no fallback. If down, ALL fund ops halt — users can't add/withdraw funds or trade.

CLASS System SPOF

5 services depend on SQL Server CLASS DB. No circuit breaker, no timeout config. If down: no statements, no ledger, no CRM, no data sync.

40+ findAll() Calls

Unbounded queries including transactionRequestRepository.findAll() in payments. Growing table + no pagination = OOM.

Timezone: ZoneId.systemDefault()

Financial date calcs use system timezone. K8s containers in UTC = IST 00:30 payment assigned to wrong business day. Affects settlement dates.

Thread Pool from User Input

RecoverOrderService: newFixedThreadPool(request.size()). Malicious request with 10K items = 10K threads = JVM crash.

Order Recon: Auto-Recovery OFF

OrderStatusReconScheduler detects mismatches but recoverOrders() is commented out. Orders filled at exchange never appear in app.
